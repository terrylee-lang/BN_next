| 欄位 | 類型 | 說明 |

| 建議標題 | 文字 | 編輯建議的中文標題，子代理動筆前應優先採用 |
| 稿件類型 | 下拉 | `即時新聞` / `深度分析` / `潤稿` / `社群貼文` / `文章查核` / `教學文` |

| 來源 2 | URL / 文字 | 第二來源（選填，同上） |
| 來源 3 | URL / 文字 | 第三來源（選填，同上） |
| 備註 | 文字 | **人類給 AI 的明確指示，動筆前必讀，內容優先於所有預設寫法** |

| 完成時間 | 日期 | AI 完成後填入當天日期 |
| 社群貼文 | 文字 | AI 產出的社群貼文內容（僅 `社群貼文` 類型填入） |
| 字數 | 數字 | 稿件全文的字數，完成後由 AI 填入 |
| 建立時間 | 日期 | 自動記錄，不需填寫 |

## 執行流程

### Step 1：掃描待處理條目

使用 `Bash` 工具直接呼叫 Notion API，以屬性條件精確篩選，不受頁面是否有內文影響。

**Token**：存於 `~/.claude/settings.local.json` 的 `NOTION_TOKEN` 環境變數，使用時以 `$NOTION_TOKEN` 帶入。

**兩種篩選模式：**

| 使用者指令 | curl 指令中的 filter |
| ----------- | --------- |
| 「跑今天的稿件」、「處理所有待辦」等通用指令 | 僅篩選 `狀態 = 未開始` |
| 「處理先泰的稿件」、「跑柔瑋的待辦」等指定負責人指令 | `狀態 = 未開始` AND `負責人 = [指定姓名]` |

**通用模式（所有未開始稿件）：**

```bash
curl -s -X POST "https://api.notion.com/v1/databases/a438232ce0a94fe6b70d9f2e9199a32a/query" \
  -H "Authorization: Bearer $NOTION_TOKEN" \
  -H "Notion-Version: 2022-06-28" \
  -H "Content-Type: application/json" \
  -d '{
    "filter": {
      "property": "狀態",
      "status": { "equals": "未開始" }
    }
  }'
```

**指定負責人模式（例如先泰）：**

```bash
curl -s -X POST "https://api.notion.com/v1/databases/a438232ce0a94fe6b70d9f2e9199a32a/query" \
  -H "Authorization: Bearer $NOTION_TOKEN" \
  -H "Notion-Version: 2022-06-28" \
  -H "Content-Type: application/json" \
  -d '{
    "filter": {
      "and": [
        { "property": "狀態", "status": { "equals": "未開始" } },
        { "property": "負責人", "select": { "equals": "先泰" } }
      ]
    }
  }'
```

負責人選項：`先泰` / `柔瑋` / `美欣` / `若彤` / `建鈞` / `祈安` / `宗穎`

API 回傳 JSON，從每筆結果的 `properties` 欄位讀取：標題（`標題.title[0].plain_text`）、負責人（`負責人.select.name`）、稿件類型（`稿件類型.select.name`）、來源（`來源 1.url`、`來源 2.url`、`來源 3.url`）、備註（`備註.rich_text[0].plain_text`）、建議標題（`建議標題.rich_text[0].plain_text`）、頁面 ID（`id`）。

篩選後列出待處理清單，並在開始前告知使用者：「找到 N 筆待處理稿件（負責人：[姓名 或 全部]），開始處理。」

若沒有符合條件的條目，直接回報：「目前無待處理稿件。」並結束流程。

### Step 2：並行處理所有條目

#### 2a. 批量標記為「進行中」

對所有待處理條目，**依序逐一**呼叫 `notion-update-page`，將每筆的 `狀態` 改為「進行中」（此步驟仍循序，避免 Notion API 頻率限制）。

#### 2b. 判斷稿件類型

**判斷優先順序：**

1. `稿件類型` 欄位有值 → 直接採用
2. `稿件類型` 為空，但 `備註` 有說明 → 從備註內容推斷
3. 兩者皆空 → 依下方判斷規則自動推斷

**各類型判斷規則：**

| 稿件類型 | 判斷依據 | 使用子代理 |
|---------|---------|-----------|
| `即時新聞` | 來源為主流媒體網址（彭博社、路透社、CNBC、紐時、The Verge、中央社、BBC 等新聞媒體） | `news-daily` |
| `深度分析` | 來源為頁數較多的 PDF 檔，或部落格平台長文（Substack、Medium 等），內容具強烈分析性或解釋性，聚焦商業模式或技術迭代洞察 | `deep-analysis` |
| `潤稿` | 來源為一篇已完整的文章（通常備註或標題已標示「潤稿」） | `draft-polish` |
| `教學文` | 來源為 X（Twitter）創作者貼文、方法論分享，或任何以「教讀者如何做某件事」為核心的工具教學素材 | `tutorial-article` |
| `社群貼文` | 來源為數位時代文章連結，任務為將內文轉化為社群貼文 | `social-post` |
| `文章查核` | 來源為一篇需核對事實的文章（日期、人物、數字、事件順序等） | `article-checker` |

**模糊情境處理：**

- **深度分析 vs 教學文難以判斷時**（例如：部落格長文同時具備分析性與教學性）→ **暫停，主動詢問使用者**，說明文章特徵並請確認要用哪種類型，不自動猜測
- **其他無法判斷的情況** → 預設使用 `news-daily`，並在頁面內文開頭標註 `[自動判斷為即時新聞，請人工確認]`

#### 2c. 並行派發所有子代理

所有條目完成類型判斷後，**在同一則訊息中一次呼叫所有 `Agent` 工具**，同時啟動所有子代理並行處理。

使用 `Agent` 工具，依判斷結果派發對應的子代理（`subagent_type` 填入對應 skill 名稱）。

**來源預讀（orchestrator 負責）：**

派發子代理前，orchestrator 先逐一檢查每個來源欄位：

| 來源格式 | 判斷方式 | 處理 |
|---------|---------|------|
| **本地路徑** | 以 `/` 或 `~/` 開頭，副檔名為 `.md` | orchestrator 用 `Read` 讀取檔案內容，將全文直接寫進子代理 prompt。若 Read 失敗（檔案不存在、路徑錯誤），視同來源失敗，記錄失敗原因，不傳給子代理 |
| **URL** | 以 `http://` 或 `https://` 開頭 | 維持原邏輯，將 URL 傳給子代理，由子代理自行 web_fetch |
| **空白** | 欄位無值 | 跳過 |

傳給子代理的 prompt 包含：
- 原始標題
- 建議標題（若有，子代理應優先以此為稿件標題）
- 來源 1（必填）、來源 2、來源 3（選填，有則一併傳入）。若為本地路徑，傳入的是 **已讀取的全文內容**（附上原始檔名供溯源），不是路徑本身
- **備註（人類指示）**：若備註欄有內容，視為編輯的明確指令，**必須在動筆前先讀完，並在整篇稿件中落實**。備註優先於所有預設寫法。

子代理負責讀取來源內容並產出完整稿件（本地來源已由 orchestrator 預讀，子代理僅需處理 URL 來源）。

**關鍵規則：來源讀取失敗時，停止而非硬寫。**

子代理的 prompt 中必須包含以下指令：

> 你必須成功讀取至少一個主要來源的完整原文，才能開始撰稿。如果所有來源都無法讀取（web_fetch 失敗、403、付費牆、逾時），**不要使用其他二手來源改寫**，直接回傳以下格式：
> `SOURCE_FAILED: [來源 URL] — [失敗原因]`
> 不產出任何稿件內容。用不完整素材硬寫會導致引言被改寫、數據無法核實等事實錯誤。

**付費牆媒體（Bloomberg、WSJ、日經、Business Insider）同樣適用此規則。** 不使用 Playwright 備援。

#### 2c-bis. 來源失敗彙整與補件

所有子代理回傳後，檢查是否有 `SOURCE_FAILED` 回報。

- **有失敗項目**：將所有失敗的稿件彙整，一次向使用者索取原文。格式如下：

```
📎 以下稿件的來源無法讀取，請提供原文（可用 Dia 瀏覽器擷取 Markdown 貼入）：

1. 《標題一》— Bloomberg 付費牆
2. 《標題二》— 來源回傳 403
```

使用者提供原文後，重新派發對應的子代理，將原文直接寫在 prompt 中傳入。

- **全部成功**：直接進入下一步。

#### 2d. 事實查核（自動套用確認修改）

所有稿件產出完成後（含補件重跑的），寫回 Notion 之前，對每篇稿件執行事實查核。

**執行方式：** 使用 `Agent` 工具，`subagent_type` 填入 `article-checker`，將完整稿件內容傳入，要求三層查核（錯字、邏輯、事實）與貨幣換算。可並行啟動所有查核代理。

**查核結果處理：**

| 查核結果類型 | 處理方式 |
|---|---|
| 確認有誤且有明確正確答案（如錯誤稱謂、錯誤名稱、引言與原文不符） | **直接套用修改** |
| 數據或引言無法從來源核實 | 在該段落後加上 `[編按：此數據/引言在標註來源中無法找到，請確認出處或考慮移除]` |
| 來源連結失效或缺失 | 在資料來源區標註 `[編按：請補完整連結]` 或 `[編按：連結已失效，建議移除]` |
| 無問題 | 不動 |

查核完成後，使用套用修改後的版本進入下一步寫回。

#### 2e. 寫回 Notion

所有稿件查核完成後，**依序逐一**寫回（此步驟仍循序，避免 Notion API 頻率限制）。每篇分兩步寫回：

**步驟一：用 `replace_content` 寫入頁面內文（稿件全文）**

稿件全文一律用 `replace_content` 寫入頁面內文，不寫進任何屬性欄位。原因：`update_properties` 對含 Markdown code block 的長文有解析問題，會觸發 JSON 錯誤。

`社群貼文` 類型例外：貼文內容仍寫入 `社群貼文` 屬性欄位（內容短、無 code block，`update_properties` 可正常處理）。

**步驟二：用 `update_properties` 寫入短欄位**

```
摘要       → 75 個全形字元以內，一句話說明稿件核心價值
重點一     → 50 字以內
重點二     → 50 字以內
重點三     → 50 字以內
字數       → 計算頁面內文的字數後填入（數字）
狀態       → 「完成」
完成時間   → 當天日期
```

**注意：Notion 流程的稿件不存入 `draft/`。** Notion 頁面即為 single source of truth，不需要本地副本。`draft/` 僅用於非 Notion 流程的稿件（Terry 臨時交辦的產稿任務）。

**若來源網址無法讀取（403、付費牆、逾時），或子代理執行中發生錯誤：**

- 用 `replace_content` 寫入：`⚠️ 處理失敗，請人工確認。（原因：[說明]）`
- `狀態` 改為「失敗」

### Step 3：完成回報與 inbox 去留確認

所有條目處理完畢後，輸出摘要：

```
✅ 今日產稿完成

共處理 N 筆：
- 《標題一》：即時新聞 → 完成（查核通過）
- 《標題二》：深度分析 → 完成（查核已修正 2 處，1 處待編輯確認）
- 《標題三》：潤稿 → 完成
```

若有查核後需人工確認的項目，另外列出：

```
📝 查核標記（編輯請留意文中 [編按] 標註）：
- 《標題二》：某數據來源待確認、某引言出處待補
```

若有失敗或需人工介入的條目，另外列出：

```
⚠️ 需人工確認（N 筆）：
- 《標題》：來源無法讀取
```

#### inbox 檔案去留確認

若本次有任何來源為 inbox/ 本地路徑，在完成回報的同時，列出所有已消化的 inbox 檔案，附上預設建議，請使用者一次確認：

**預設建議邏輯：**

| 稿件類型 | 預設建議 | 理由 |
|---------|---------|------|
| 即時新聞 | 刪除 | 一次性事件，原始 URL 保留在 Notion 來源欄位 |
| 深度分析 | 移至 research/ | 公司／技術主題有累積價值 |
| 教學文 | 保留 inbox | 方法論素材可能再用 |
| 潤稿 | 刪除 | 草稿已轉為正式稿 |
| 社群貼文 | 刪除 | 來源文章已發布 |
| 文章查核 | 刪除 | 查核完成即結案 |

**輸出格式：**

```
📂 inbox 檔案去留：
1. OpenAI memo - The Verge.md → 建議刪除（即時新聞，已消化）
2. OpenAI memo - Gizmodo.md → 建議刪除（同上）
3. AI權力地圖報告.md → 建議移至 research/（深度分析素材）
4. Prompt教學法.md → 建議保留（教學文素材）

請確認，或逐項調整（例如「3 留 inbox」「1 移 research」）。
```

使用者確認後執行。移至 research/ 的檔案套用 research 標準格式，更新 `research/_index.md`。

**與 Wiki Ingest 的分工**：若 inbox 檔案已確認移至 research/，該篇素材的 Wiki Ingest 不再另建 research 筆記（避免重複）。Wiki Ingest 仍可為同一篇稿件涉及的**其他**公司、人物、技術主題建立獨立筆記。
