## 執行流程

### Step 1：掃描待處理條目

使用 `Bash` 工具透過 Notion API 查詢資料庫，**不要使用 notion-fetch**（notion-fetch 對資料庫只會回傳 schema，無法列出條目）。

#### 查詢指令範本

**情境 A：指定負責人**（使用者說「跑先泰的稿件」）
```bash
curl -s \
  -X POST \
  -H "Authorization: Bearer $NOTION_TOKEN" \
  -H "Notion-Version: 2022-06-28" \
  -H "Content-Type: application/json" \
  -d '{
    "filter": {
      "and": [
        {"property": "狀態", "status": {"equals": "未開始"}},
        {"property": "負責人", "select": {"equals": "先泰"}}
      ]
    },
    "sorts": [{"property": "建立時間", "direction": "ascending"}]
  }' \
  "https://api.notion.com/v1/databases/a438232ce0a94fe6b70d9f2e9199a32a/query"
```

**情境 B：不指定負責人**（使用者說「跑所有待辦」）
```bash
curl -s \
  -X POST \
  -H "Authorization: Bearer $NOTION_TOKEN" \
  -H "Notion-Version: 2022-06-28" \
  -H "Content-Type: application/json" \
  -d '{
    "filter": {"property": "狀態", "status": {"equals": "未開始"}},
    "sorts": [{"property": "建立時間", "direction": "ascending"}]
  }' \
  "https://api.notion.com/v1/databases/a438232ce0a94fe6b70d9f2e9199a32a/query"
```

從 JSON 回應中提取每筆條目的：`id`（頁面 ID）、`標題`、`稿件類型`、`來源 1/2/3`、`備註`、`負責人`。

列出符合條件的清單，按 `建立時間` 升序排列（先建立的先處理）。

若沒有任何待處理條目，直接回報：「今日無待處理稿件。」並結束流程。

---

### Step 2：逐條依序處理

對每一筆待處理條目，依序執行以下步驟（**不要並行**，避免 Notion API 頻率限制）：

#### 2a. 標記為「進行中」

使用 `notion-update-page` 將該條目的 `狀態` 改為「進行中」。

#### 2b. 判斷稿件類型

**判斷優先順序：**

1. `稿件類型` 欄位有值 → 直接採用
2. `稿件類型` 為空，但 `備註` 有說明 → 從備註內容推斷
3. 兩者皆空 → 依下方判斷規則自動推斷

**各類型判斷規則：**

| 稿件類型 | 判斷依據 | 使用子代理 |
|---------|---------|-----------|
| `即時新聞` | 來源為主流媒體網址（彭博社、路透社、CNBC、紐時、The Verge、中央社、BBC 等新聞媒體） | `news-daily` |
| `深度分析` | 來源為頁數較多的 PDF 檔，或部落格平台長文（Substack、Medium 等），內容具強烈分析性或解釋性，聚焦商業模式或技術迭代洞察 | `deep-analysis` |
| `潤稿` | 來源為一篇已完整的文章（通常備註或標題已標示「潤稿」） | `draft-polish` |
| `教學文` | 來源為 X（Twitter）創作者貼文、方法論分享，或任何以「教讀者如何做某件事」為核心的工具教學素材 | `tutorial-article` |
| `社群貼文` | 來源為數位時代文章連結，任務為將內文轉化為社群貼文 | `social-post` |
| `文章查核` | 來源為一篇需核對事實的文章（日期、人物、數字、事件順序等） | `article-checker` |

**模糊情境處理：**

- **深度分析 vs 教學文難以判斷時**（例如：部落格長文同時具備分析性與教學性）→ **暫停，主動詢問使用者**，說明文章特徵並請確認要用哪種類型，不自動猜測
- **其他無法判斷的情況** → 預設使用 `news-daily`，並在產出內容開頭標註 `[自動判斷為 daily 稿，請人工確認]`

#### 2c. 派發子代理

使用 `Agent` 工具，依判斷結果派發對應的子代理（`subagent_type` 填入對應 skill 名稱）。

傳給子代理的 prompt 包含：
- 原始標題
- 來源 1（必填）、來源 2、來源 3（選填，有則一併傳入）
- **備註（人類指示）**：若備註欄有內容，視為編輯的明確指令，**必須在動筆前先讀完，並在整篇稿件中落實**。備註優先於所有預設寫法。

子代理負責讀取來源內容並產出完整稿件。

#### 2d. 寫回 Notion

子代理完成後，使用 `notion-update-page` 將結果寫入該條目：
- `重點摘要`：三大核心洞察，格式為 `1. xxx\n2. xxx\n3. xxx`，每點 50 字以內
- `產出內容`：所有類型的主稿全文一律填入此欄
- `社群貼文`：僅在備註中明確要求同時產出社群貼文時才填入；純社群貼文任務的正文仍填 `產出內容`
- `字數`：計算 `產出內容` 的字數後填入
- `狀態`：改為「完成」
- `完成時間`：填入當天日期

**若來源網址無法讀取（403、付費牆、逾時），或子代理執行中發生錯誤：**
- `產出內容` 填入：`⚠️ 處理失敗，請人工確認。（原因：[說明]）`
- `狀態` 改為「失敗」（讓你一眼識別需要人工介入的條目）

---

### Step 3：完成回報

所有條目處理完畢後，輸出摘要：

```
✅ 今日產稿完成

共處理 N 筆：
- 《標題一》：即時新聞 → 完成
- 《標題二》：深度分析 → 完成
- 《標題三》：潤稿 → 完成
```

若有失敗或需人工介入的條目，另外列出：

```
⚠️ 需人工確認（N 筆）：
- 《標題》：來源無法讀取
```
